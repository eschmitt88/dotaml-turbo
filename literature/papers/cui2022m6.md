# M6-Rec: Generative Pretrained LMs as Open-Ended Recommenders

## TL;DR

Alibaba's M6-Rec converts every recommendation task (retrieval,
ranking, CTR, explanation, conversational rec, etc.) into a text task
on top of M6, an existing GPT/T5-style language model pre-trained on
web corpora + user-behavior plain-text representations. The lever is
not the architecture — M6 itself is unmodified — but the *task
unification template*: any rec task is reframed as language
understanding or language generation by serializing user features,
candidate features, and cross features into natural-language segments.
For efficient deployment they introduce **option tuning** (a prompt-
tuning variant with frozen base, 1% trainable params) and **multi-
segment late interaction** for low-latency CTR (cache the early `L'`
layers' outputs per segment, only run the last `L − L'` layers at
request time).

## Claims

- **A single pretrained generative LM can serve as the foundation for
  retrieval, ranking, zero-shot recommendation, explanation,
  personalized content creation, and conversational rec** — the user
  behavior data is just serialized to text and the same M6 backbone
  handles all tasks (Section 1, Section 4).
- **Option tuning beats fine-tuning** at 1% trainable parameters by
  reusing the last few soft-prompt embeddings as the softmax
  classification layer (Figure 2). This combines with adapter tuning
  to outperform full fine-tuning while preserving the base model for
  multi-task serving.
- **Multi-segment late interaction makes Transformer-LM CTR
  prediction practical for real-time serving.** By splitting the
  user/candidate text into multiple segments and pre-computing the
  first `L'` layers' representations per segment, M6-Rec caches the
  static parts and only computes the final `L − L'` layers per
  request, drastically cutting inference latency (Figure 3).
- **Text-based serialization sidesteps item-ID embeddings** and
  enables zero-shot recommendation in new domains (the system can
  recommend items it has never seen, because they're identified by
  their text descriptions).
- **Avoiding catastrophic forgetting** is the main practical reason
  for prompt/option tuning over full fine-tuning when one model must
  serve many tasks.

## Methods

The base M6 is a Transformer LM trained with a text-infilling objective
where the input is split into a bidirectional region and an
autoregressive region (Figure 1). User behavior is serialized as plain
text like "A male user in Beijing, who clicked product X last night
and product Y this noon, was recommended product Z and did not click
it." Task-specific adaptation is via option tuning: insert trainable
soft prompts + treat the last few as classifier embeddings, train only
these (1% params), freeze the base. For latency-sensitive CTR, the
text is decomposed into segments (user-static, user-recent-history,
candidate-features), each pre-encoded through `L'` Transformer layers
offline; at request time, only the segments' top `L − L'` layers run.

## Takeaways for foundation-mvp-740

- **The headline transferable idea: "one encoder, many task-as-prompt
  heads".** For us this means a single Transformer encoder over the
  10-hero + player-feature draft, with **multiple lightweight heads
  for win / duration / item-recommendation / KDA / first-blood /
  tower-state**, all sharing the same encoder. We already do this in
  [[2026-05-20-rich-supervision-multitask-740]] but the M6 framing
  argues for *prompt-tokens-as-task-id* rather than a fixed set of
  head classes — useful when we want to add a 6th, 7th head later
  without re-architecting.
- **DO NOT adopt the text-serialization route.** Our setting is
  tabular-structured (10 hero IDs + 80 player features); converting
  to natural language adds tokens, latency, and noise for no benefit.
  M6-Rec's text design choice is a constraint of using a pretrained
  text LM as the spine, which we are NOT doing. The "task unification
  via prompts" pattern is what transfers; the "everything is text"
  part does not.
- **Option tuning / adapter pattern is the right idea for the
  fine-tuning stage.** Pretrain the foundation model once with the MAE
  + contrastive objective; fine-tune per-task with frozen base +
  per-task adapter (or with task-prefix tokens, which is the cleanest
  version of "option tuning" in our setting). This enables many task
  heads at low marginal cost and prevents the win head's gradients
  from destroying the duration head's adapter.
- **Late interaction is interesting for `dotaml-serve` (the live
  inference companion repo), not `dotaml-turbo`.** Worth flagging in
  the proposal's "future work" — at serving time, the foundation model
  could pre-encode the per-account history segments offline and only
  run the top layers per request. Defer.
- **Use this paper as the rhetorical frame** when justifying
  foundation-mvp-740 to a future reader: M6-Rec proves the
  "one-foundation-many-tasks" pattern works in production
  recommendation; we're applying the same template to draft-time win
  prediction, with tabular structure instead of text.

## Open questions / caveats

- M6-Rec relies on M6 having been pre-trained on web corpora — the
  zero-shot recommendation magic comes from the LM's world knowledge,
  not the foundation architecture itself. We have no equivalent
  pre-trained backbone for Dota drafts; foundation-mvp-740 is starting
  from scratch on Turbo data. The "open-ended" claim does not
  transfer.
- The paper's evaluation is product-team-internal benchmarks (Taobao,
  Alipay) with anonymized metrics; some claims are hard to verify
  rigorously. The architectural ideas (option tuning, late
  interaction, task-as-prompt) are well-described and reproducible
  independently of the specific numbers.
- M6-Rec is *very* much a recommendation paper; it does NOT address
  the inductive biases for predicting a discrete competitive outcome
  (win/lose). For us, the win head is still a CLS-style classification
  — the M6 lessons are about *organizing* the head set, not designing
  any individual head.
- The 1% trainable params claim relies on M6 being huge (multi-B
  params); our foundation will be small (5M target). Option tuning's
  parameter savings are less compelling at small scale.
